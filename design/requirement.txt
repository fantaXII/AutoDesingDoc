<<자동 요약 system>>
여러개의 git hub
각 git hub는 
  < code로 된 git hub인 경우>
    - folder별로 디자인 문서가 있고, 프로젝트 전체에 대한 디자인 문서가 있음
    - 다중 폴더가 있을 경우 file이 25개 이상 있는 폴더에대해서만 디자인 문서를 만든다.
    - 각 git hub의 file이 update될때마다, 해당 파일이 포함된 디자인 문서를 자동으로 업데이트한다.
      - 예를 들어 folder A에 30개의 파일. A folder의 하위 폴더로 AA folder가 있고 거기에 5개의 파일이 있으면, folder A에 20개, AA folder의 5개 파일의 내용까지 합쳐서 A folder에 A_design.md 문서가 나와야해.
      - 만약 디자인 문서가 있던 폴더의 파일이 25개 에서 더 줄더라도 10개 이하가 되기 전까지는 design문서를 유지해줘.
      - 10개 이하가 되면 해당 디자인 문서를 상위 디자인 문서에 합치고, 삭제해줘.
      - 상위 design문서와 하위 design문서는 서로를 찾아 갈 수 있도록 cross reference section도 있어야해.
      - 만약 folder Z 가 5개이 파일이 있고, 그 하위에 folder Y(33 개 file),  folder W(file 8 개)가 있는 경우,  folder Y에 design 문서가 하나 있어야 하고, folder W는 design 문서가 없어야 하며, folder Z에는 folder z,y,w 모두에 대한 내용이 포함한 design 문서가 있어야해. 이 문서에 folder y 관련된 내용은 folder z의 입장에서 구조를 표시 하기에 적절안 정도만 있으면 되고, 자세한 내용은 folder y의 design문서에 포함되면 되어.
    - 25개의 파일 수를 셀 때 node_modules, .gitignore, 이미지 파일, 바이너리 파일 등은 제외해야 합니다. 
    - 최소 25개~수십 개의 파일 코드를 한 번에 LLM에 넣으면 Context Window(토큰 제한)를 초과할 수 있습니다.
      "개별 파일을 먼저 짧게 요약하고, 그 요약본들을 모아서 폴더 단위 디자인 문서를 생성(Map-Reduce 방식)하도록 파이프라인을 구성하는게 좋을 것 같아.
    - 업데이트 시간, 트리거 파일 등의 정보를 "Markdown의 YAML Frontmatter 형식으로 최상단에 기록하는 게 좋을 것 같아
    - 생성하는 design문서는 타겟 폴더 안에 생성해줘.

==> 디자인 문서인 만큼 code나 value는 포함하지 말고, 개념과 enum등의 의미등, 목적,구조와 시나리오 위주로 작성되면 되어
==> 나중에 lint와 같이 검증을 하기 위해, 마지막 update가 언제인지, 어떤 file때문인지등도 있어야 할것 같아.
==> git hub외에도 perforce CLI로도 확장 할 수 있도록 해줘. (GitHub Webhook/API와 Perforce CLI를 모두 지원할 수 있도록 VCS Adapter 또는 Interface 패턴을 적용해서 설계)

2. 추천하는 프롬프트 수정본 (Copy & Paste 용)
위의 보완점들을 반영하여, LLM이 구조적으로 이해하기 쉽도록 카테고리화한 수정 버전입니다. 필요에 따라 입맛에 맞게 수정해서 사용해 보세요.

[역할 및 목적]
당신은 복잡한 다중 저장소(Multi-Repository) 환경에서 코드베이스를 분석하고 디자인 문서를 자동 생성/유지보수하는 '자동 요약 시스템'을 구현하는 시니어 소프트웨어 엔지니어입니다.
아래의 요구사항을 충족하는 안정적이고 확장 가능한 시스템의 코드를 작성해 주세요.

[시스템 아키텍처 및 확장성]
현재는 대상이 여러 개의 GitHub 저장소이지만, 향후 Perforce CLI 등 다른 VCS로도 확장할 수 있어야 합니다. (VCS Provider Interface/Adapter 패턴을 적용하세요).
소스 코드 변경을 감지하는 트리거 메커니즘을 고려하여 엔트리 포인트를 설계하세요.

[문서 생성 및 계층화 규칙 (File Thresholds)]
분석 대상: 프로젝트 전체 디자인 문서 1개, 그리고 조건에 맞는 폴더별 디자인 문서를 생성합니다. (이미지, 바이너리, 의존성 폴더 등은 파일 카운트에서 제외하는 필터링 로직 포함)
생성 조건: 특정 폴더(하위 폴더 포함) 내의 유효 파일 개수가 총 25개 이상일 때만 해당 폴더에 디자인 문서(폴더명_design.md)를 생성합니다.

유지 및 삭제 (Hysteresis):
이미 디자인 문서가 생성된 폴더는, 파일 수가 25개 미만으로 떨어지더라도 10개 이하가 되기 전까지는 문서를 계속 유지 및 업데이트합니다.
파일 수가 10개 이하로 떨어지면, 해당 디자인 문서의 핵심 내용을 상위 폴더의 디자인 문서로 병합(Merge)한 뒤 해당 문서를 삭제합니다.

계층 구조 처리 (Cross-Reference):
예) Z 폴더(파일 5개), 하위 Y 폴더(33개), 하위 W 폴더(8개)일 경우:
총합 46개이므로 Z 폴더에는 디자인 문서 생성. (Z, Y, W 내용 모두 포함)
Y 폴더는 33개이므로 자체 디자인 문서 생성.
W 폴더는 8개이므로 문서를 생성하지 않음.
이때 Z 폴더의 문서 내에 작성되는 Y 폴더 관련 내용은 'Z의 관점에서 본 Y의 구조/역할' 정도로만 간략히 요약하고, 상세 내용은 Y 폴더의 디자인 문서를 링크(Cross Reference)하여 서로 찾아갈 수 있게 구성하세요.

[문서 내용 및 포맷팅 (Content Rule)]
내용 제한: 디자인 문서이므로 실제 소스 코드 블록이나 Raw Value는 포함하지 마세요.
포함 요소: 개념, Enum 등의 의미, 주요 목적, 디렉토리 구조, 그리고 핵심 시나리오 워크플로우 위주로 작성해야 합니다.
대규모 파일 처리 시 LLM의 컨텍스트 한계를 고려하여, 개별 파일을 먼저 요약한 뒤 이를 취합해 문서화하는 파이프라인(Map-Reduce 등) 구조를 고려해 코드를 작성하세요.

[메타데이터 및 검증 (Metadata)]
향후 Linter 등에서 문서를 검증할 수 있도록, 마크다운 파일 최상단에 YAML Frontmatter를 사용하여 다음 정보를 반드시 기록하세요.
last_updated: 마지막 업데이트 시간
trigger_file: 이번 업데이트를 발생시킨 원인 파일명 (또는 커밋 해시)
total_files: 현재 폴더 구조에서 카운트된 총 파일 수

이 정도 구체적인 제약사항과 아키텍처 패턴을 함께 제시하면, LLM이 단순히 스크립트 수준이 아니라 확장성과 예외 처리가 고려된 탄탄한 객체지향/모듈화 코드를 뱉어낼 확률이 훨씬 높아집니다. 추가로 궁금한 점이 있으신가요?

아래 요구사항과 아키텍처 가이드라인을 충족하는 안정적이고 확장 가능한 시스템 코드를 작성해 주세요.

<<자동 요약 시스템 요구사항 및 아키텍처 가이드>>

1. 시스템 아키텍처 및 확장성 (VCS & AI 생태계 연동)
- VCS Provider 추상화: 현재 타겟은 여러 개의 GitHub 저장소이지만, 향후 Perforce CLI 등 다른 시스템으로 쉽게 확장할 수 있도록 Interface/Adapter 패턴(`VCSAdapter`)을 적용하세요.
- AI 컨텍스트 최적화: 생성된 문서들은 향후 MCP(Model Context Protocol) 서버를 통해 AI 에이전트가 즉각적으로 읽을 수 있는 형태로 제공될 수 있어야 합니다. 
- 글로벌 컨텍스트 관리: 전체 프로젝트의 핵심 구조와 룰을 묶어주는 상위 컨텍스트 파일(`CLAUDE.md` 등)과 개별 폴더의 디자인 문서가 유기적으로 연결되도록 구조화하세요.

2. 파일 필터링 및 트리거 최적화
- 대상 필터링: 코드 라인 수 산정 시 `node_modules`, `.venv`, `.gitignore`에 포함된 항목, 빌드 산출물, 이미지/바이너리 파일 등은 제외하도록 설정 파일에서 관리하게 하세요.
- 지능형 트리거 (비용 최적화): 파일이 업데이트될 때마다 무조건 LLM을 호출하지 않습니다. Git Diff나 AST(Abstract Syntax Tree) 분석을 통해 클래스/함수 시그니처, Public Interface, Enum 등 구조적으로 유의미한 변경이 발생했을 때만 문서 업데이트 파이프라인을 트리거하도록 구현하세요.

3. 문서 생성 및 계층화 규칙 (Hysteresis 알고리즘)
- 생성 임계값: 폴더(하위 폴더 포함, 필터링 후 유효 파일 기준) 내 파일 개수가 '25개 이상'일 때 해당 폴더에 디자인 문서(`폴더명_design.md`)를 생성합니다.
- 유지/삭제 임계값 (Hysteresis): 파일 수가 25개 미만으로 줄어도 즉각 삭제하지 않고 유지합니다. 파일 수가 '10개 이하'로 떨어지는 시점에 해당 디자인 문서를 삭제하되, 삭제 전 핵심 내용을 상위 폴더의 디자인 문서로 병합(Merge)하세요.
- Cross-Reference (계층 구조): 
  (예: Z 폴더 하위에 파일 5개, 하위 Y 폴더 파일 33개, 하위 W 폴더 파일 8개)
  - Z 폴더: 총합 46개이므로 디자인 문서 생성 (Z, Y, W 내용 포함)
  - Y 폴더: 33개이므로 자체 디자인 문서 생성
  - W 폴더: 8개이므로 문서 미생성
  - 상/하위 문서 연결: Z 폴더 문서 내의 Y 폴더 관련 내용은 'Z 관점에서의 구조와 역할' 정도로만 요약하고, 상세 내용은 Y 폴더 디자인 문서로 하이퍼링크(Cross-Reference)를 걸어 서로 참조 가능하게 하세요.

4. LLM 파이프라인 및 문서 내용 포맷팅
- Map-Reduce 처리: 여러 파일의 코드를 한 번에 처리하면 LLM 토큰 제한(Context Window)에 걸릴 수 있습니다. 개별 파일을 짧게 요약한 뒤, 이를 모아서 폴더 단위 디자인 문서로 합성하는 Map-Reduce 방식의 파이프라인을 구성하세요.
- 내용 제약 (No Raw Code): 실제 소스 코드 블록이나 Raw Value는 포함하지 마세요. 개념, Enum 등의 의미, 핵심 목적, 디렉토리 구조 및 주요 시나리오 워크플로우 위주로 요약해야 합니다.

5. 메타데이터 및 지식 관리 연동
- 생성되는 모든 마크다운 파일 최상단에 YAML Frontmatter를 추가하세요. 향후 Obsidian(Dataview 플러그인 등)이나 Linter에서 동적으로 쿼리하고 검증할 수 있어야 합니다.
- 필수 메타데이터:
  - `last_updated`: 마지막 업데이트 시간 (ISO 8601)
  - `trigger_file`: 문서 업데이트를 발생시킨 원인 파일명 또는 커밋 해시
  - `total_files`: 현재 폴더 구조에서 카운트된 유효 파일 수
  - `status`: 현재 상태 (예: active, merging, archived)

위 명세에 따라, 핵심 비즈니스 로직(VCS 연동, 트리거 감지, Hysteresis 폴더 카운팅, LLM 요약 파이프라인)을 처리하는 시스템 구조와 주요 클래스/함수의 코드를 작성해 주세요.
